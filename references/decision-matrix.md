# NPI门禁决策矩阵与遗留问题机制

> Agent 的**推理知识**：用于结合用户实际情况判断"放行 / 有条件放行 / 不放行"及遗留问题走向。**不要原样整段丢给用户**，要落到用户具体情境里解释。

## 一、放行决策矩阵

| 条件 | 决策 | 说明 |
|------|------|------|
| 全部达标 | ✅ 放行 | 直接进入下一阶段 |
| 遗留问题≤3项，无安全/法规类 | ⚠️ 有条件放行 | 需质量总监批准，每周跟踪 |
| 安全/法规类遗留 | ❌ 不放行 | 立即整改 |
| 遗留问题>5项 | ❌ 不放行 | 需重新评审 |

**关键判定口径（结合用户具体信息时用）：**
- "安全/法规类遗留"一票否决，无论数量多少→不放行。
- "有条件放行"的硬前提：**无安全/法规类遗留**且**遗留≤3项**（部分阶段允许≤5项，见各阶段放行条件）。
- 各阶段放行硬指标不同（如 PVT 要求 Cpk≥1.33、试产合格率≥98%），需对照该阶段门禁检查表判定，而非只看遗留问题数量。

## 二、有条件放行审批流程

```
遗留问题评估
    ↓
问题数量≤3？ → 否 → 不放行
    ↓是
无安全/法规类？ → 否 → 不放行
    ↓是
填写有条件放行申请表
    ↓
质量总监批准
    ↓
指定关闭时间和责任人
    ↓
每周跟踪汇报
    ↓
全部关闭后确认
```

## 三、遗留问题追踪机制

**追踪频率**：每周跟踪一次，直到全部关闭

**升级规则**：
- 超过计划关闭时间1周 → 质量经理知会
- 超过计划关闭时间2周 → 质量总监介入
- 超过计划关闭时间1个月 → 项目暂停

**关闭确认（需同时满足）**：
- 问题解决有证据
- 改善措施有效
- 预防措施已落实

## 四、跨阶段传递原则
- 每个阶段门禁的"遗留问题"必须明确责任人、计划关闭时间，并传递到下一阶段继续跟踪。
- EVT→DVT 遗留关闭率要求≥90%；DVT→PVT 要求全部关闭；PVT→MP 要求全部关闭。
- "带病转移"是高频翻车点：引导用户关注前阶段遗留是否**真正关闭**，而非口头"差不多"。
